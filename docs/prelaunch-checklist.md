# アンテナサイト 公開直前チェックリスト

---

## 1. 法務 / 規約

- [ ] 利用規約・プライバシーポリシーのページが公開URLで表示される
- [ ] Cookieバナー・同意取得フローが法的要件（個人情報保護法・GDPR該当国向け）を満たしている
- [ ] 外部コンテンツ（RSS/スクレイピング）の引用・転載ルールを確認し、著作権侵害リスクがないと法務が承認している
- [ ] アフィリエイト・広告表示が景品表示法上の「広告である旨」を明示している
- [ ] フィードから取得する情報に個人情報（氏名・メアド等）が混入しないよう除外処理が実装済み
- [ ] お問い合わせフォームに個人情報の取り扱い同意チェックボックスがある
- [ ] 特定商取引法の適用可否を確認し、必要な場合は表示ページを作成済み
- [ ] 未成年利用制限・年齢確認が必要なカテゴリに適切な制限が設けられている
- [ ] 各記事ページに元サイトへの被リンクが存在し、出典表示が適切である
- [ ] ドメイン登録者情報（Whois）の公開範囲を意図どおりに設定済み

---

## 2. 技術 / 計測

- [ ] 本番環境でHTTPS（TLS 1.2以上）が有効、HTTP→HTTPSリダイレクトが動作している
- [ ] Core Web Vitals（LCP / FID / CLS）が目標値（LCP ≤ 2.5s / CLS ≤ 0.1）を満たしている
- [ ] フィード取得クローラーが本番スケジュール（5〜15分間隔等）で正常動作している
- [ ] robots.txt が意図した設定（クローラー許可/禁止）になっており、sitemap.xmlへの参照がある
- [ ] Google Analytics 4 / Search Console の計測タグが全ページで発火していることをデバッガーで確認
- [ ] CDNキャッシュ・サーバーキャッシュのTTLが記事更新頻度と整合している
- [ ] データベースバックアップが自動実行され、リストア手順を1名以上が実施確認している
- [ ] エラー監視（Sentry等）のアラート通知先（Slack/メール）が設定済みで疎通確認済み
- [ ] サーバー死活監視（UptimeRobot等）が稼働し、ダウン時のアラートが届くことを確認
- [ ] 負荷テスト（想定ピークの1.5倍）を実施し、レスポンスタイム・エラー率が許容範囲内

---

## 3. コンテンツ品質

- [ ] 全カテゴリでフィードが正常に取得・表示されており、文字化け・レイアウト崩れがない
- [ ] 画像が存在しない記事のサムネイルにデフォルト画像が正しく表示される
- [ ] 外部リンクがすべて `target="_blank" rel="noopener noreferrer"` になっている
- [ ] OGP（og:title / og:image / og:description）がトップ・カテゴリ・記事ページで正しく出力される
- [ ] 記事タイトル・ディスクリプションにSQLインジェクション・XSSが入らないよう sanitize 処理が確認済み
- [ ] 重複コンテンツ対策として canonical タグが正しく設定されている
- [ ] 404ページ・500ページがカスタムデザインで表示され、離脱を防ぐナビゲーションがある
- [ ] スパム・アダルト・暴力コンテンツのフィルタリングロジックが動作し、誤検知率が許容範囲内
- [ ] モバイル表示（iOS Safari / Android Chrome）で主要ページが崩れなく表示される
- [ ] アクセシビリティ：img に alt が設定され、キーボードナビゲーションが基本動作する

---

## 4. 審査運用

- [ ] 新規フィード申請フローの受付窓口（フォーム or メール）が稼働し、自動返信メールが送られる
- [ ] フィード審査基準（ジャンル・更新頻度・品質スコア閾値）がドキュメント化されている
- [ ] 審査担当者と承認権限者が明確に割り当てられ、バックアップ担当も指定済み
- [ ] 審査ステータス（申請中・承認・否認・保留）を管理する台帳またはツールが整備されている
- [ ] 否認フィードへの通知・理由説明フローが定義され、テンプレート文が用意されている
- [ ] 既存掲載フィードの定期再審査（月次等）スケジュールが決定している
- [ ] ユーザー通報機能が実装され、通報→確認→対処のSLAが定義されている（例：24h以内確認）
- [ ] 掲載停止・緊急削除の実行権限者と手順（管理画面操作 or APIコール）が文書化されている
- [ ] 審査ログが90日以上保存され、監査時に検索・出力できる
- [ ] 審査担当者向け操作マニュアルが最新版にアップデートされ、全員が参照可能な場所にある

---

## 5. インシデント対応

- [ ] インシデントレベル定義（P0〜P3等）と対応SLA（P0：15分以内着手等）が合意されている
- [ ] オンコール担当スケジュール（ローンチ後72h）と連絡先リストが周知されている
- [ ] フィード取得停止・DB接続断・高負荷の各障害シナリオに対応したRunbookが存在する
- [ ] 障害発生時のユーザー向け告知方法（サイト内バナー / SNS / ステータスページ）が決定している
- [ ] 個人情報漏洩が疑われる場合の報告フロー（72h以内の監督官庁届出を含む）が整備されている
- [ ] スパム・不正アクセス検知時のIPブロック手順がRunbookに記載されている
- [ ] フィードから有害コンテンツが流入した場合の緊急非表示オペが訓練済みである
- [ ] データベースロールバック手順が文書化され、ステージング環境で実施確認済み
- [ ] インシデント終了後のポストモーテム（振り返り）テンプレートが用意されている
- [ ] 過去のステージング障害・リハーサルで洗い出した課題がすべてクローズされている

---

## 6. ローンチ判定（Go / No-Go）

- [ ] 上記1〜5のチェックリストで「必須」ラベル付き項目が全てチェック済みである
- [ ] ステージング環境での総合リハーサルが直近48h以内に完了し、クリティカルバグがゼロ
- [ ] 関係部門（開発・法務・編集・マーケ・CS）の責任者全員が公開承認に署名している
- [ ] 広告ネットワーク・アフィリエイトASPの審査が通過し、掲載コードが本番に組み込まれている
- [ ] SNSアカウント・メルマガ等の告知チャネルが公開タイミングに合わせてスタンバイしている
- [ ] GAでリアルタイム計測が確認でき、ローンチ後のKPIダッシュボードが閲覧可能な状態にある
- [ ] No-Go条件（P0相当バグ残存・法務未承認・フィード0件等）が発動していないことを確認
- [ ] ローンチ後72hのサポート体制（問い合わせ対応・監視当番）が人員・ツールともに準備済み
- [ ] ロールバック判断基準（例：エラー率5%超・主要フィード全滅等）と実行手順が合意されている
- [ ] 最終Go/No-Go判定会議を実施し、プロダクトオーナーが「Go」を宣言している

## 公開当日オペ（時系列）

### T-24h（前日）

- [ ] 本番環境への最終デプロイを完了し、デプロイログを保存する
- [ ] ステージングで全カテゴリ・全デバイスの表示確認を実施、スクリーンショットを記録する
- [ ] 監視アラートのテスト発報を行い、全担当者に通知が届くことを確認する
- [ ] フィード取得クローラーを本番スケジュールで起動し、初回取得データを目視確認する
- [ ] 広告タグ・ASPコードが本番ページに正しく表示されていることを確認する
- [ ] ローンチ当日の連絡体制（Slackチャンネル・電話ツリー）を全員に通知する

### T-3h（公開3時間前）

- [ ] 本番サーバーのリソース（CPU/MEM/Disk）が正常範囲内であることを確認する
- [ ] キャッシュをフラッシュし、最新コンテンツが反映されることを確認する
- [ ] SNS告知・プレスリリースの予約投稿が正しい時刻にセットされていることを確認する
- [ ] オンコール担当者が待機状態にあることをSlackで確認する

### T0（公開）

- [ ] DNSまたはロードバランサーの切り替え・設定変更を実施し、公開URLでアクセスできることを確認する
- [ ] GAリアルタイムでアクセスが計測されていることを確認する（0件なら即調査）
- [ ] トップページ・カテゴリページ・記事ページを複数端末で目視確認する
- [ ] SNS告知を投稿し、リンク先が正しく開くことを確認する
- [ ] インシデント対応Slackチャンネルを「監視中」ステータスに設定する

### T+6h（公開6時間後）

- [ ] エラーログ・アラート履歴を確認し、P0/P1インシデントが発生していないことを確認する
- [ ] GAでセッション数・直帰率・ページ/セッションが想定レンジ内であることを確認する
- [ ] 広告・アフィリエイトのインプレッション・クリックデータが計上されていることを確認する
- [ ] ユーザー通報・問い合わせフォームの受信内容を確認し、緊急対応が必要なものを処理する
- [ ] サーバーリソースが高負荷になっていないかチェックし、必要ならスケールアウトを実施する

### T+24h（公開翌日）

- [ ] 24hのKPI（PV / UU / 滞在時間 / CTR / 広告収益）を集計し、目標比を記録する
- [ ] フィード取得エラー率・失敗フィード一覧を確認し、上位件数を修正または削除する
- [ ] ユーザーフィードバック（通報・問い合わせ）の初期集計をまとめ、改善タスクを登録する
- [ ] ローンチ後24h振り返りミーティングを実施し、想定外事象をポストモーテムに記録する
- [ ] 次のデプロイサイクル・改善スプリントのスケジュールを確定し、関係者に共有する
