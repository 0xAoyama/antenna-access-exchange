# Day30 Gate1 判定シート（供給成立）

バージョン: 1.0
対象期間: Day1-30

## 1. KPI定義

KPI-A 許諾済みソース数
定義: 利用同意が完了し、ACTIVEのソース総数
閾値: 20以上
数式: 許諾済みソース数 = COUNT(ステータス = ACTIVE)

KPI-B 更新達成率
定義: 計画更新ジョブのうち期限内に正常完了した割合
閾値: 90.0%以上
数式: 更新達成率 = (正常完了ジョブ数 ÷ 計画ジョブ総数) × 100

## 3. 自動判定ロジック

判定条件
KPI-A >= 20 かつ KPI-B >= 90.0 なら PASS
KPI-A < 20 かつ KPI-B >= 90.0 なら FAIL-A
KPI-A >= 20 かつ KPI-B < 90.0 なら FAIL-B
KPI-A < 20 かつ KPI-B < 90.0 なら FAIL-AB

判定記入欄

KPI-B実績: ______ %
最終判定: PASS / FAIL-A / FAIL-B / FAIL-AB

## 4. FAIL時の是正アクション

### FAIL-A（ソース数不足）

| No | アクション | 担当 | 期限 |
|---|---|---|---|
| A-1 | 未許諾ソースを理由別に分類（交渉中/審査待ち/技術待ち） | 渉外 | 翌営業日 |
| A-2 | 交渉中ソースへ督促送信 | 渉外 | 翌営業日 |
| A-3 | 代替ソースを優先順位で追加申請 | 渉外 | Day33 |
| A-4 | 中間再判定を実施 | Gate管理者 | Day35 |

不足数の式
不足ソース数 = 20 - KPI-A実績値

### FAIL-B（更新達成率不足）

| No | アクション | 担当 | 期限 |
|---|---|---|---|
| B-1 | 失敗ジョブの原因分類（NW/認証/形式/タイムアウト） | インフラ | 翌営業日正午 |
| B-2 | 分類別の恒久対策を立案 | インフラ | Day32 |
| B-3 | タイムアウト対策（間隔/バッチサイズ調整） | インフラ | Day33 |
| B-4 | 認証情報更新（APIキー/トークン） | データ管掌 | Day32 |
| B-5 | 7日再計測して再判定 | Gate管理者 | Day42 |

再判定基準
Day35-42の7日間達成率 >= 90.0%

### FAIL-AB（両方未達）
FAIL-AとFAIL-Bを並行実施
Day31にステアリングへエスカレーション
